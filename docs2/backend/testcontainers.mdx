# Testcontainers

## Introduction

Testcontainers is a library that makes it easy to run real development and test dependencies in disposable Docker containers.

It helps developers create reliable, repeatable integration tests by bootstrapping required services such as databases, message brokers, and caches on demand instead of relying on shared or mocked infrastructure.

Using Testcontainers, you can keep tests consistent across local development and CI, automatically wait for containers to become ready, and ensure cleanup of container resources after test execution.


## Problems Testcontainers solve

- **Infrastructure setup is brittle and hard to replicate**. Developers must manually provision and configure the services their app depends on, which leads to inconsistent local and CI environments.
- **Shared dependencies cause non-deterministic tests**. When databases, brokers, or caches are reused across users or pipelines, test results can vary because of data pollution and configuration drift.
- **Mocks and in-memory substitutes differ from production**. Fake services often lack real behavior, features, or initialization semantics, so tests can miss integration issues.
- **Docker and Docker Compose require too much manual coordination**. Directly managing containers or compose files may lead to port conflicts, race conditions, and unreliable service readiness during test execution.
- **Resource cleanup is error-prone**. Ensuring containers and related resources are removed after tests, even when tests fail, is difficult without automation.

## Benefits of using Testcontainers

- **On-demand isolated test infrastructure**. Services are provisioned dynamically in fresh Docker containers for each test run, avoiding shared state and reducing environmental drift.
- **Consistency between local and CI environments**. The same containerized dependencies can be used in development machines and continuous integration pipelines, making test outcomes more reliable.
- **Real service behavior instead of mocks**. Tests run against actual databases, message brokers, and other infrastructure, which improves fidelity compared to in-memory or stubbed alternatives.
- **Automatic readiness handling**. Built-in wait strategies ensure containers are fully initialized before tests interact with them, reducing flaky startup timing issues.
- **Cleanup automation**. Testcontainers manages stopping and removing containers, volumes, and networks after tests complete, even when failures occur.


## Usage

A typical Testcontainers-based integration test works as follows:

- **Before test execution**: start required services as Docker containers via the Testcontainers API, then configure your app to use those containerized dependencies.
- **During test execution**: run tests against the real containerized services.
- **After test execution**: let Testcontainers automatically stop and remove containers, even if tests fail.


## Getting Started

Use Testcontainers to test a Node.js application with PostgreSQL:

1. Install dependencies:

```bash
npm install pg --save
npm install jest @testcontainers/postgresql --save-dev
```

2. Write a test that starts PostgreSQL in a container, connects with `pg`, and runs against real database state:

```javascript
const { Client } = require("pg");
const { PostgreSqlContainer } = require("@testcontainers/postgresql");
const { createCustomerTable, createCustomer, getCustomers } = require("./customer-repository");

describe("Customer Repository", () => {
  jest.setTimeout(60000);

  let postgresContainer;
  let postgresClient;

  beforeAll(async () => {
    postgresContainer = await new PostgreSqlContainer().start();
    postgresClient = new Client({ connectionString: postgresContainer.getConnectionUri() });
    await postgresClient.connect();
    await createCustomerTable(postgresClient);
  });

  afterAll(async () => {
    await postgresClient.end();
    await postgresContainer.stop();
  });

  it("should create and return customers", async () => {
    await createCustomer(postgresClient, { id: 1, name: "John Doe" });
    const customers = await getCustomers(postgresClient);
    expect(customers).toEqual([{ id: 1, name: "John Doe" }]);
  });
});
```

3. Run the test with `npm test`. Testcontainers will start PostgreSQL for the test, let the test connect to the real service, and clean up containers afterward.


## Wait Strategies

Testcontainers wait strategies let your test code pause until a container is actually ready before interacting with it.

Reference: https://node.testcontainers.org/features/wait-strategies

### Listening ports (default)

Waits until mapped ports are bound.

Example:

```javascript
const { GenericContainer } = require("testcontainers");

const container = await new GenericContainer("alpine")
  .withExposedPorts(6379)
  .start();
```

### Log output

Waits until the container logs a specific message.

Example:

```javascript
const { GenericContainer, Wait } = require("testcontainers");

const container = await new GenericContainer("alpine")
  .withWaitStrategy(Wait.forLogMessage("Ready to accept connections"))
  .start();
```

### Health check

Waits until the container’s built-in Docker health check succeeds.

Example:

```javascript
const { GenericContainer, Wait } = require("testcontainers");

const container = await new GenericContainer("alpine")
  .withHealthCheck({
    test: ["CMD-SHELL", "curl -f http://localhost || exit 1"],
    interval: 1000,
    timeout: 3000,
    retries: 5,
    startPeriod: 1000,
  })
  .withWaitStrategy(Wait.forHealthCheck())
  .start();
```

### HTTP

Waits for an HTTP endpoint to return a valid response.

Example:

```javascript
const { GenericContainer, Wait } = require("testcontainers");

const container = await new GenericContainer("redis")
  .withWaitStrategy(Wait.forHttp("/health", 8080))
  .start();
```

### Shell command

Waits until a command inside the container exits successfully.

Example:

```javascript
const { GenericContainer, Wait } = require("testcontainers");

const container = await new GenericContainer("alpine")
  .withWaitStrategy(Wait.forSuccessfulCommand("stat /tmp/app.lock"))
  .start();
```

### One shot

For containers that run briefly and exit with code `0`.

Example:

```javascript
const { GenericContainer, Wait } = require("testcontainers");

const container = await new GenericContainer("alpine")
  .withWaitStrategy(Wait.forOneShotStartup())
  .start();
```

### Composite

Combine multiple strategies with `Wait.forAll([...])`.

Example:

```javascript
const { GenericContainer, Wait } = require("testcontainers");

const container = await new GenericContainer("alpine")
  .withWaitStrategy(Wait.forAll([
    Wait.forListeningPorts(),
    Wait.forLogMessage("Ready to accept connections"),
  ]))
  .start();
```


## Containers

### Start a container

Create and start any container with `GenericContainer`:

```javascript
const { GenericContainer } = require("testcontainers");

const container = await new GenericContainer("alpine").start();
```

Use a specific image version:

```javascript
const container = await new GenericContainer("alpine:3.10").start();
```

### Image pull policy

By default Testcontainers pulls images if needed. You can force a pull:

```javascript
const { GenericContainer, PullPolicy } = require("testcontainers");

const container = await new GenericContainer("alpine")
  .withPullPolicy(PullPolicy.alwaysPull())
  .start();
```

### Configure startup behavior

- `withCommand(["sleep", "infinity"])`
- `withEntrypoint(["cat"])`
- `withEnvironment({ ENV: "VALUE" })`
- `withPlatform("linux/arm64")`

### File handling

Copy files, directories, content, or archives into the container before it starts:

```javascript
const container = await new GenericContainer("alpine")
  .withCopyFilesToContainer([{ source: "/local/file.txt", target: "/remote/file1.txt" }])
  .withCopyDirectoriesToContainer([{ source: "/localdir", target: "/some/nested/remotedir" }])
  .withCopyContentToContainer([{ content: "hello world", target: "/remote/file2.txt" }])
  .withCopyArchivesToContainer([{ tar: nodeReadable, target: "/some/nested/remotedir" }])
  .withCopyToContainerOptions({ copyUIDGID: true })
  .start();
```

You can also copy after startup:

```javascript
const container = await new GenericContainer("alpine").start();

await container.copyFilesToContainer([{ source: "/local/file.txt", target: "/remote/file1.txt" }]);
await container.copyContentToContainer([{ content: "hello world", target: "/remote/file2.txt" }]);
await container.copyArchiveToContainer(nodeReadable, "/some/nested/remotedir", { copyUIDGID: true });
```

### Container metadata

- `withLabels({ label: "value" })`
- `withName("custom-container-name")` (not recommended)
- `withWorkingDir("/opt")`
- `withDefaultLogDriver()`
- `withTmpFs({ "/temp_pgdata": "rw,noexec,nosuid,size=65536k" })`
- `withUser("bob")`

### Security and privileges

- `withPrivilegedMode()`
- `withSecurityOpt("no-new-privileges")`
- `withAddedCapabilities("NET_ADMIN", "IPC_LOCK")`
- `withDroppedCapabilities("NET_ADMIN", "IPC_LOCK")`
- `withUlimits({ memlock: { hard: -1, soft: -1 } })`
- `withIpcMode("host")`

### Resources and host settings

- `withResourcesQuota({ memory: 0.5, cpu: 1 })`
- `withSharedMemorySize(512 * 1024 * 1024)`
- `withCustomHostname("my-hostname")` (not recommended)

### Managing life cycle

Stop a container quickly:

```javascript
await container.stop();
```

Wait for stop completion:

```javascript
await container.stop({ timeout: 10_000 });
```

Keep container after stop:

```javascript
await container.stop({ remove: false });
```

Disable auto-cleanup for a specific container:

```javascript
const container = await new GenericContainer("alpine")
  .withAutoCleanup(false)
  .start();
```

### Reuse containers

Reuse a container across tests:

```javascript
const container1 = await new GenericContainer("alpine")
  .withCommand(["sleep", "infinity"])
  .withReuse()
  .start();

const container2 = await new GenericContainer("alpine")
  .withCommand(["sleep", "infinity"])
  .withReuse()
  .start();

expect(container1.getId()).toBe(container2.getId());
```

### Exposing ports

Expose ports and retrieve mapped host ports:

```javascript
const container = await new GenericContainer("alpine")
  .withExposedPorts(80)
  .start();

const httpPort = container.getMappedPort(80);
```

For a single port, use:

```javascript
const httpPort = container.getFirstMappedPort();
```

You can specify protocols:

```javascript
.withExposedPorts({ container: 80, protocol: "udp" })
```

### Running commands

Execute commands inside a running container:

```javascript
const { output, stdout, stderr, exitCode } = await container.exec(["echo", "hello", "world"]);
```

With options:

```javascript
const result = await container.exec(["echo", "hello", "world"], {
  workingDir: "/app/src/",
  user: "1000:1000",
  env: { VAR1: "enabled" }
});
```

### Streaming logs

Read logs from a started container:

```javascript
(await container.logs())
  .on("data", line => console.log(line))
  .on("err", line => console.error(line));
```

Or attach a consumer before start:

```javascript
const container = await new GenericContainer("alpine")
  .withLogConsumer(stream => {
    stream.on("data", line => console.log(line));
  })
  .start();
```


## Images

### Build an image from a Dockerfile

Use `GenericContainer.fromDockerfile()` to build and start a custom image:

```javascript
const { GenericContainer } = require("testcontainers");

const container = await GenericContainer
  .fromDockerfile("/path/to/build-context")
  .build();

const startedContainer = await container.start();
```

### Keep the built image

By default the built image is removed on exit. Preserve it with a name and `deleteOnExit: false`:

```javascript
const container = await GenericContainer
  .fromDockerfile("/path/to/build-context")
  .build("my-custom-image", { deleteOnExit: false });
```

### Build options

- `withBuildArgs({ ARG: "VALUE" })`
  ```javascript
  const container = await GenericContainer
    .fromDockerfile("/path/to/build-context")
    .withBuildArgs({ ARG: "VALUE" })
    .build();
  ```

- `withTarget('my-stage')`
  ```javascript
  const container = await GenericContainer
    .fromDockerfile("/path/to/build-context")
    .withTarget('my-stage')
    .build();
  ```

- `withCustomDockerfile("/path/to/build-context", "my-dockerfile")`
  ```javascript
  const container = await GenericContainer
    .fromDockerfile("/path/to/build-context", "my-dockerfile")
    .build();
  ```

- `withPlatform("linux/amd64")`
  ```javascript
  const container = await GenericContainer
    .fromDockerfile("/path/to/build-context")
    .withPlatform("linux/amd64")
    .build();
  ```


## Networking

### Create and use a Docker network

Create a network and start containers inside it:

```javascript
const { GenericContainer, Network } = require("testcontainers");

const network = await new Network().start();

const container = await new GenericContainer("alpine")
  .withNetwork(network)
  .start();

await container.stop();
await network.stop();
```

Get a container’s IP within that network:

```javascript
const networkIpAddress = container.getIpAddress(network.getName());
```

### Network mode

Set Docker network mode directly:

```javascript
const container = await new GenericContainer("alpine")
  .withNetworkMode("bridge")
  .start();
```

:::note
`host` mode works only on Linux.
:::


### Network aliases

Use aliases for container-to-container communication on the same network:

```javascript
const network = await new Network().start();

const container = await new GenericContainer("alpine")
  .withCommand(["sleep", "infinity"])
  .withNetwork(network)
  .start();

const fooContainer = await new GenericContainer("alpine")
  .withCommand(["sleep", "infinity"])
  .withNetwork(network)
  .withNetworkAliases("foo")
  .start();

expect((await container.exec(["getent", "hosts", "foo"])).exitCode).toBe(0);
```

### Expose host ports to containers

Expose a local host port so containers can reach it via `host.testcontainers.internal`:

```javascript
const { GenericContainer, TestContainers } = require("testcontainers");
const { createServer } = require("http");

const server = createServer((req, res) => {
  res.writeHead(200);
  res.end("hello world");
});
server.listen(8000);

await TestContainers.exposeHostPorts(8000);

const container = await new GenericContainer("alpine")
  .withCommand(["sleep", "infinity"])
  .start();

expect((await container.exec(["curl", `http://host.testcontainers.internal:8000`])).output)
  .toBe("hello world");
```

This works by starting an SSHd container and mapping the exposed host port through it so other containers can access it.


## Compose Guide

### Start a Docker Compose environment

Use `DockerComposeEnvironment` with a compose path and file:

```js title="compose.js"
const { DockerComposeEnvironment } = require("testcontainers");

const composeFilePath = "/path/to/build-context";
const composeFile = "docker-compose.yml";

const environment = await new DockerComposeEnvironment(composeFilePath, composeFile).up();
```

Start only selected services:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .up(["redis", "postgres"]);
```

### Wait strategy

Use `withWaitStrategy()` with Docker container names, not service names. With Compose v2, first replica names are usually `<service>-1`.

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withWaitStrategy("redis-1", Wait.forLogMessage("Ready to accept connections"))
  .withWaitStrategy("postgres-1", Wait.forHealthCheck())
  .up(["redis", "postgres"]);
```

Override the default wait strategy for all services:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withDefaultWaitStrategy(Wait.forHealthCheck())
  .up();
```

### Pull policy

By default, images are pulled if missing. Override it:

```js title="compose.js"
const { DockerComposeEnvironment, PullPolicy } = require("testcontainers");

const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withPullPolicy(PullPolicy.alwaysPull())
  .up();
```

Custom pull policy example:

```js title="compose.js"
const { GenericContainer, ImagePullPolicy } = require("testcontainers");

class CustomPullPolicy implements ImagePullPolicy {
  public shouldPull(): boolean {
    return true;
  }
}

const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withPullPolicy(new CustomPullPolicy())
  .up();
```

### Build and environment settings

Rebuild images before starting:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withBuild()
  .up();
```

Use a custom `.env` file:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withEnvironmentFile(".env.custom")
  .up();
```

Enable Compose profiles:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withProfiles("profile1", "profile2")
  .up();
```

Disable recreate mode:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withNoRecreate()
  .up();
```

Set environment variables for compose substitution:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withEnvironment({ "TAG": "VALUE" })
  .up();
```

### Project name and cleanup

Use a custom project name:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withProjectName("test")
  .up();
```

Disable automatic cleanup registration while keeping explicit teardown available:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withProjectName("test")
  .withAutoCleanup(false)
  .up();
```

### Custom Docker Compose client

Provide custom compose client options:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile)
  .withClientOptions({ executable: { standalone: true, executablePath: "/path/to/docker-compose" } })
  .up();
```

### Downing and stopping environments

Tear down the environment:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile).up();
await environment.down();
```

Wait for shutdown with timeout:

```js title="compose.js"
await environment.down({ timeout: 10_000 });
```

Keep volumes after teardown:

```js title="compose.js"
await environment.down({ removeVolumes: false });
```

Stop instead of down when sharing networks or dependencies:

```js title="compose.js"
const environment = await new DockerComposeEnvironment(composeFilePath, composeFile).up();
await environment.stop();
```

### Interacting with compose containers

Access containers by their compose-managed name format `<service-name>-<index>`:

```js title="compose.js"
const container = environment.getContainer("alpine-1");
```

This guide covers the main Compose options available in `testcontainers-node`, including startup, wait handling, pull policy, rebuild, environment customization, cleanup control, and container access.
