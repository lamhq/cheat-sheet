# Workspace

## Overview

PNPM Workspace is a feature that allows you to manage multiple interdependent project (packages or apps) within a single repository (aka monorepo).

It's ideal for projects that contain multiple apps, libraries, or services that share dependencies or tooling.


## Workspace protocol

The workspace protocol (`workspace:`) is a special way to declare dependencies inside a pnpm monorepo.

It ensures that dependencies resolve strictly to local workspace packages instead of falling back to the npm registry.

How it works:
1. PNPM uses a `pnpm-workspace.yaml` file at the root of your repo to define which projects belong to your workspace.
2. Each project has its own `package.json`, and dependencies between them can be linked using the `workspace:` protocol.
3. When you publish, `workspace:` specs in `package.json` are automatically converted into semver ranges.

Example:

```json title="package.json"
{
  "dependencies": {
    "foo": "workspace:*",        // Always link to local foo
    "bar": "workspace:^1.0.0",   // Link only if compatible version exists locally
    "baz": "workspace:../baz",   // Reference by relative path
    "qar": "workspace:^"         // Convert to caret range on publish
  }
}
```

## Example

Given the following monorepo structure:

```
my-monorepo/
├── pnpm-workspace.yaml
├── packages/
│   └── shared-utils/
│       └── package.json
└── apps/
    └── web-app/
        └── package.json
```

```yaml title="pnpm-workspace.yaml"
packages:
  - "packages/*"
  - "apps/*"
```

`package.json` of the `shared-utils` project:

```json title="packages/shared-utils/package.json"
{
  "name": "shared-utils",
  "version": "1.0.0",
  "main": "index.js"
}
```

`package.json` of the `web-app` project:

```json title="apps/web-app/package.json"
{
  "name": "web-app",
  "version": "1.0.0",
  "dependencies": {
    "shared-utils": "workspace:*"
  }
}
```

- The `web-app` project declares `shared-utils` as a dependency using the **`workspace:*`** protocol.  
- `workspace:*`: always link to the local `shared-utils` project, regardless of version.  
- If `shared-utils` didn't exist locally, installation would fail instead of fetching from npm.
- On publish: pnpm converts `workspace:*` into the actual version (`"1.0.0"`) in the published `package.json`.


## Configuring Workspace

Create a `pnpm-workspace.yaml` file in the project root to define the projects in the workspace:

```yaml title="pnpm-workspace.yaml"
packages:
  - "apps/*"
  - "packages/*"
```
- `apps/`: contains the main applications of the project.
- `packages/`: contains shared libraries or utilities used by the applications.


## Adding projects

To add a new project to the workspace, create a new folder under `apps/` with a `package.json` file.

For example, to add a `web` project:
```
.
├── apps
│   └── web
│       └── package.json
├── package.json
├── pnpm-lock.yaml
└── pnpm-workspace.yaml
```


## Installing dependencies

Install a package to a project in the workspace:

```bash
# install `lodash` for `web` project
pnpm --filter web add lodash
pnpm -F web add lodash
```

Remove a dependency from a project:

```bash
# remove `lodash` from `web` project
pnpm -F web remove lodash
```

Install a dependency into the root of the monorepo (root `package.json`):

```bash
# install `eslint` to the root `package.json`
pnpm add eslint -w
```

:::info
The workspace project name is defined in project's `package.json` (`name` field).
:::


## Running scripts

Run a script (defined in the `package.json`) of a project:

```bash
pnpm -F <project> run <script-name>
```

Run a script across **all projects** in the workspace:

```bash
pnpm -r run <script-name>
```

Run script in all projects whose name has prefix `@repo/`:
```bash
pnpm -F '@repo/**' run <script-name>
```

Run in projects that depend on `utils`:
```bash
pnpm -F '...utils' run <script-name>
```


## Running binaries

Run a local installed binary from a specific project:

```bash
# run Playwright tests in the `web-e2e` project
pnpm -F web-e2e exec playwright test
```
