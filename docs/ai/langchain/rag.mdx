# Retrieval-Augmented Generation

## Overview

Large Language Models (LLMs) have two key limitations:
- **Finite context**: they can’t ingest entire corpora at once.
- **Static knowledge**: their training data is frozen at a point in time.

**Retrieval** solves this by fetching relevant external knowledge at query time, forming the foundation of **Retrieval-Augmented Generation (RAG)**.

## Building a Knowledge Base

A **knowledge base** is a repository of documents or structured data used during retrieval.

If you need a custom knowledge base, you can use LangChain’s document loaders and vector stores to build one from your own data.

If you already have a knowledge base (e.g., a SQL database, CRM, or internal documentation system), you can connect it as a tool for an agent in Agentic RAG.

### Retrieval Pipeline

Retrieval allows LLMs to access relevant context at runtime.

The retrieval pipeline becomes a foundation for a broader system that combines search with generation.

A typical retrieval workflow looks like this:

### Building Blocks

- **Document Loaders**: Ingest data from external sources (Google Drive, Slack, Notion)
- **Text Splitters**: Break documents into retrievable chunks
- **Embedding Models**: Convert text into vectors with similar semantics near each other
- **Vector Stores**: Specialized databases for storing and searching embeddings
- **Retrievers**: Interfaces that return documents given a query


## RAG Architectures

RAG can be implemented in multiple ways:

| Architecture    | Description                      | Control   | Flexibility | Latency     |
| --------------- | -------------------------------- | --------- | ----------- | ----------- |
| **2-Step RAG**  | Retrieve before generating       | ✅ High   | ❌ Low      | ⚡ Fast     |
| **Agentic RAG** | LLM decides when/how to retrieve | ❌ Low    | ✅ High     | ⏳ Variable |
| **Hybrid**      | Combines both with validation    | ⚖️ Medium | ⚖️ Medium   | ⏳ Variable |


### 2-Step RAG

Retrieval always happens before generation. Straightforward, predictable, and suitable for FAQs and documentation bots.

### Agentic RAG

An agent decides **when** and **how** to retrieve information instead of retrieving documents before answering.

The agent uses tools (documentation loaders, web APIs, databases) to fetch external knowledge.

```mermaid theme={"theme":{"light":"catppuccin-latte","dark":"catppuccin-mocha"}}

    D --> H{Enough to answer?}
    H -- No --> B
    H -- Yes --> I[Generate final answer]
```

```python
import requests
from langchain.tools import tool
from langchain.chat_models import init_chat_model
from langchain.agents import create_agent

@tool
def fetch_url(url: str) -> str:
    """Fetch text content from a URL"""
    response = requests.get(url, timeout=10.0)
    response.raise_for_status()
    return response.text

system_prompt = """\
Use fetch_url when you need to fetch information from a web-page; quote relevant snippets.
"""

agent = create_agent(
    model="claude-sonnet-4-6",
    tools=[fetch_url], # A tool for retrieval
    system_prompt=system_prompt,
)
```


### Hybrid RAG

Combines both approaches with intermediate steps such as query preprocessing, retrieval validation, and post-generation checks.

These systems offer more flexibility (than fixed pipelines) while maintaining some control over execution.

Typical components include:
- **Query Enhancement**: Rewrite/expand query for better retrieval
- **Retrieval Validation**: Check if docs are sufficient; refine if needed
- **Answer Validation**: Check if answer is grounded and complete; regenerate if needed

- Systems needing quality control
