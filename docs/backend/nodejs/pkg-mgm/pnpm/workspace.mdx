# Workspace

## Overview

PNPM Workspace is a feature that allows you to manage multiple interdependent packages within a single repository (monorepo).

It's ideal for projects that contain multiple apps, libraries, or services that share dependencies or tooling.


## Workspace protocol

The workspace protocol (`workspace:`) is a special way to declare dependencies inside a pnpm monorepo.

It ensures that dependencies resolve strictly to local workspace packages instead of falling back to the npm registry.

How it works:
1. PNPM uses a `pnpm-workspace.yaml` file at the root of your repo to define which folders contain packages.
2. Each package has its own `package.json`, and dependencies between them can be linked using the `workspace:` protocol.
3. When you publish, `workspace:` specs in `package.json` are automatically converted into semver ranges.

Example:

```json
{
  "dependencies": {
    "foo": "workspace:*",        // Always link to local foo
    "bar": "workspace:^1.0.0",   // Link only if compatible version exists locally
    "baz": "workspace:../baz",   // Reference by relative path
    "qar": "workspace:^"         // Convert to caret range on publish
  }
}
```

## Example

Given the following monorepo structure:

```
my-monorepo/
├── pnpm-workspace.yaml
├── packages/
│   └── shared-utils/
│       └── package.json
└── apps/
    └── web-app/
        └── package.json
```

```yaml title="pnpm-workspace.yaml"
packages:
  - "packages/*"
  - "apps/*"
```

`package.json` of the `shared-utils` package:

```json title="packages/shared-utils/package.json"
{
  "name": "shared-utils",
  "version": "1.0.0",
  "main": "index.js"
}
```

`package.json` of the `web-app` application:

```json title="apps/web-app/package.json"
{
  "name": "web-app",
  "version": "1.0.0",
  "dependencies": {
    "shared-utils": "workspace:*"
  }
}
```

Here, the app declares `shared-utils` as a dependency using the **`workspace:*`** protocol.  
- `workspace:*` means: always link to the local `shared-utils` package, regardless of version.  
- If `shared-utils` didn't exist locally, installation would fail instead of fetching from npm.
- On publish: pnpm converts `workspace:*` into the actual version (`"1.0.0"`) in the published `package.json`.


## Configuring Workspace

Create a `pnpm-workspace.yaml` file in the project root to define the packages in the workspace:

```yaml title="pnpm-workspace.yaml"
packages:
  - "apps/*"
  - "packages/*"
```
- `apps/`: contains the main applications of the project.
- `packages/`: contains shared libraries or utilities used by the applications.


## Adding applications

To add a new application to the workspace, create a new folder under `apps/` with a `package.json` file.

For example, to add a `web` application:
```
.
├── apps
│   └── web
│       └── package.json
├── package.json
├── pnpm-lock.yaml
└── pnpm-workspace.yaml
```

## Adding packages

To add a new package to the workspace, create a new folder under `packages/` with a `package.json` file.

For example, to add a `utils` package:
```
.
├── packages
│   └── utils
│       └── package.json
├── package.json
├── pnpm-lock.yaml
└── pnpm-workspace.yaml
```


## Installing dependencies

Install a npm package to a repo in the workspace:

```bash
pnpm -F <app-name> add <npm-package> -D
```
- `-F` (or `--filter`) run command in specific projects
- `--recursive` run command in every project of a workspace

Install a shared package into an application:

```bash
pnpm -F <app-name> add <package-name> --workspace
```

You can also specify the version range of the dependencies:
```bash
pnpm -F <app-name> add <package-name>@^1.0.0 --workspace
```

> [!NOTE]
> The version is defined using the `version` field in `package.json` of the package.

> [!INFO]
> `app-name` and `package-name` are defined using the `name` field in `package.json`


## Running scripts

Run a script of a specific package:

```bash
pnpm -F <package-name> run <script-name>
```

Run a script across **all packages** in the workspace:

```bash
pnpm -r run <script-name>
```

Run in all packages whose name has prefix `@repo/`:
```bash
pnpm -F '@repo/**' run dev
```

Run in packages that depend on `utils`:
```bash
pnpm -F '...utils' run build
```

Run a local installed binary from a specific package:

```bash
pnpm -F <package-name> exec jest
```
