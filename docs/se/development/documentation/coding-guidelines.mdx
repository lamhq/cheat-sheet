# Coding Guidelines

## Error Handling  
- **Guideline**: Use centralized error handling and avoid swallowing exceptions.  
- **Rationale**: Ensures consistent error reporting and easier debugging.  
- **Example**: Throw domain-specific exceptions and let the application layer handle them.

## Validation  
- **Guideline**: Perform input validation at boundaries (API endpoints, UI forms).  
- **Rationale**: Prevents invalid data from propagating through the system.  
- **Example**: Validate request payloads using a schema before passing them to business logic.

## Database Access  
- **Guideline**: Access the database only through repositories or data access services.  
- **Rationale**: Abstracts persistence logic and avoids leaking SQL/ORM details into business logic.  
- **Example**: `OrderService` calls `OrderRepository` instead of writing queries directly.

## Configuration Management  
- **Guideline**: Store configuration in environment variables or config files, not hardcoded in code.  
- **Rationale**: Improves portability and security.  
- **Example**: Database connection strings should be loaded from environment variables.

## Logging  
- **Guideline**: Use a standardized logging framework and consistent log levels.  
- **Rationale**: Ensures observability and easier troubleshooting.  
- **Example**: Use `INFO` for normal operations, `WARN` for recoverable issues, `ERROR` for failures.

## Security Practices  
- **Guideline**: Always sanitize inputs, use parameterized queries, and enforce authentication/authorization at service boundaries.  
- **Rationale**: Protects against common vulnerabilities (SQL injection, XSS, unauthorized access).  
- **Example**: API endpoints must check user roles before executing sensitive operations.
